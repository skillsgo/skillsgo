# SkillsGo Monorepo Release Design

## Goals

SkillsGo maintains the Flutter desktop App and Go Registry in one repository while preserving independent release cadences.

- A single product repository with independent release pipelines.
- Tags are the only entry point for production releases.
- Pull requests validate but never publish.
- `main` may produce temporary artifacts but never overwrite a release.
- Every production artifact resolves to one commit.
- Prefer GitHub-native permissions and GHCR over long-lived credentials.
- Do not present an unsigned and unnotarized App as a distributable production release.

The standalone CLI now exists as a third release unit. Its final public release automation is intentionally deferred until the bundled-App contract stabilizes; production App builds compile a matching CLI from the same commit.

## Release Units

| Unit | Source | Tag | Version source | Production artifacts |
| --- | --- | --- | --- | --- |
| App | `app/` | `app/vX.Y.Z` | `app/pubspec.yaml` | Signed and notarized macOS DMG plus checksums |
| CLI | `cli/` | `cli/vX.Y.Z` | tag | Deferred standalone binary matrix |
| Registry | `registry/` | `registry/vX.Y.Z` | tag | Linux and macOS binaries, checksums, and GHCR image |

A repository-wide `vX.Y.Z` tag is not used because it cannot identify the released unit and would force unrelated components to share a version.

## Version Rules

All release units follow Semantic Versioning.

```text
app/v0.1.0
cli/v0.1.0
registry/v0.1.0
```

1. A production tag must reference a commit already reachable from `main`.
2. The App tag version must match the version in `pubspec.yaml`.
3. The App build number is generated by CI and does not appear in the Git tag.
4. Registry version and build date are injected into build metadata.
5. Artifact names use normalized versions without the unit prefix.
6. Existing tags, releases, container tags, and artifacts are never overwritten.

## Pull Request CI

Every pull request runs the currently supported checks for App, CLI, and Registry. Workflow-level path filters are avoided while the repository is small so required checks never remain pending because a workflow was skipped.

```text
App       → flutter analyze + flutter test
CLI       → gofmt + go test
Registry  → gofmt + go test -race + registry build
```

CI receives read-only repository permissions. Third-party actions are pinned to full commit SHAs and updated through Dependabot.

## Registry Snapshots

Registry changes on `main`, or a manual dispatch, publish temporary images:

```text
ghcr.io/skillsgo/registry:main
ghcr.io/skillsgo/registry:sha-<short-sha>
```

Snapshots never create GitHub Releases and never write `latest`.

## Registry Production Release

A `registry/vX.Y.Z` tag reruns the complete Registry test suite and builds with a GitHub Actions matrix:

```text
linux/amd64
linux/arm64
darwin/amd64
darwin/arm64
```

Expected artifacts:

```text
skillsgo-registry_<version>_linux_amd64.tar.gz
skillsgo-registry_<version>_linux_arm64.tar.gz
skillsgo-registry_<version>_darwin_amd64.tar.gz
skillsgo-registry_<version>_darwin_arm64.tar.gz
checksums.txt
```

Stable releases update full, minor, major, and `latest` container tags. Pre-releases publish only the complete version tag.

## App Candidate Builds

Manual dispatch and App pull-request smoke tests may produce expiring, explicitly unsigned GitHub Actions artifacts. Candidate builds do not create GitHub Releases.

The candidate build compiles the bundled SkillsGo CLI for the App architecture from the same commit and verifies its version and executable contract before packaging.

## App Production Release

An `app/vX.Y.Z` release remains disabled until all of the following steps are available:

1. Validate the tag against `pubspec.yaml`.
2. Build a Universal App or explicit arm64 and x64 artifacts.
3. Build and embed the matching SkillsGo CLI.
4. Sign with a Developer ID Application certificate.
5. Submit to the Apple Notary Service.
6. Staple the notarization ticket.
7. Package the DMG.
8. Verify Gatekeeper acceptance.
9. Generate SHA-256 checksums.
10. Create the GitHub Release.

## Permissions and Credentials

Registry publication prefers `GITHUB_TOKEN` with narrowly scoped permissions for release contents, packages, OIDC, and attestations. Images publish to GHCR rather than Docker Hub.

App release credentials live in an approval-protected `app-release` GitHub Environment:

```text
APPLE_DEVELOPER_ID_CERTIFICATE
APPLE_DEVELOPER_ID_CERTIFICATE_PASSWORD
APPLE_TEAM_ID
APPLE_API_KEY_ID
APPLE_API_ISSUER_ID
APPLE_API_PRIVATE_KEY
```

Fork pull requests never receive release secrets.

## Supply Chain and Traceability

The first release phase provides:

- SHA-256 checksums;
- source commit SHA and build time;
- immutable Git tags and GitHub Releases;
- GHCR digests;
- SPDX or CycloneDX SBOMs;
- GitHub Artifact Attestations.

A later phase may add keyless container signing and a signed App-update manifest.

## Changelog

Each release unit generates notes only from changes that affect that unit. Shared documentation and root configuration appear in a unit's notes only when behavior or delivery for that unit changes.

GitHub-generated release notes are sufficient initially. A changeset or conventional-commit workflow may be introduced after release volume justifies it.

## Protocol Compatibility

App, CLI, and Registry versions do not need to match. Compatibility is expressed through protocol and capability versions rather than inferred from product SemVer.

```text
GET /v1/capabilities
```

The response includes at least a protocol version, Registry version, and feature list. Clients enable behavior based on the declared protocol and capabilities.

## Legacy Athens Release Configuration

Nested Athens workflows, Docker Hub assumptions, and GoReleaser configuration are not part of the active monorepo pipeline. Historical configuration should be removed after the replacement workflows are validated; Git history remains the archive.

The Registry uses a native GitHub Actions build matrix instead of GoReleaser because prefixed Go Module tags in this monorepo require GoReleaser Pro support. The native matrix is free, explicit, and auditable.

## Delivery Phases

### Phase 1: Registry

- Keep all required checks active.
- Build the four-platform Registry matrix.
- Publish snapshots and tag-driven releases.
- Generate checksums, an SBOM, and attestations.
- Remove obsolete Athens release configuration after dry-run validation.

### Phase 2: App Candidates

- Build the App and bundled CLI together.
- Produce unsigned macOS candidate artifacts.
- Confirm Universal Binary or dual-architecture delivery.
- Finalize the product name and bundle identifier.

### Phase 3: App Production

- Configure Apple credentials and environment approval.
- Complete signing, notarization, DMG, and Gatekeeper verification.
- Add a signed automatic-update manifest.
- Enable tag-driven App production releases.

### Phase 4: Standalone CLI

- Stabilize the App-to-CLI JSON protocol.
- Define the standalone four-platform binary matrix and install channels.
- Enable `cli/vX.Y.Z` only after compatibility and upgrade policy are documented.
