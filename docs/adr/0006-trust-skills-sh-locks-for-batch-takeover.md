# Trust skills.sh locks for batch takeover

SkillsGo offers one Library Batch Takeover action for currently discovered External Installations. The selected Library location is the authorization and scan boundary: All Skills selects User Scope and every accessible Added Project, Global selects only User Scope, and a Project route selects only that Workspace Scope. The App sends these scopes explicitly, and the CLI reads only their supported skills.sh locks. An installed Skill represented by a selected lock is trusted and taken over directly without a Hub or content match. Because these locks record one source entry per Skill but do not persist copy mode or general installation paths, the CLI scans known Agent Skill directories inside the same selected scopes and associates every discovered physical copy or symlink with the trusted lock entry.

The Library runs this validation before confirmation as an independent CLI preflight and exposes the result to App callers as a Batch Takeover Plan. The plan returns one exact eligible count for All Skills, User Scope, and every accessible Added Project, plus an opaque ID bound to the validated candidate paths, complete normalized lock identities, aliases, and filesystem-state digests. One eligible item means one physical Skill group within one declaration scope, so All Skills equals User Scope plus the per-Project counts and the confirmed count uses the same unit as the execution result. Preflight does not write the Store, Workspace Manifest, Workspace Sum, Installation Receipts, or any Agent target; its only local output is the ephemeral authorization plan under the SkillsGo cache. Plans expire after 30 minutes, and cleanup retains at most 32 current plans including the new preflight, so repeated background validation cannot grow the cache without bound. The current-location action and accessible Project rail entries display counts from that same plan rather than estimating them from visible External entries; the fixed All Skills and Global rail labels remain unadorned.

User data preservation and SkillsGo metadata integrity take precedence over making those installations uniform. Batch Takeover captures each stable physical directory, computes both the normalized Hub Content Digest and a complete filesystem-state digest covering modes and empty directories, verifies the resulting immutable Store artifact and metadata, and then reuses the normal copy-installation commit path without materializing content back over the existing target. A lock supplies trusted source identity but never proves a Hub artifact version, so these digests define a captured local baseline rather than impersonating an unverified Hub version. Baselines deduplicate only when both content and recoverable filesystem state are identical, distinct copied directories remain independent Installation Targets even when they came from one lock entry, and filesystem aliases of one symlink-resolved directory form one physical target group. The portable requirement remains copy-mode, while each exact Installation Receipt preserves whether its current target path is the physical copied directory or a symlink alias; this is the same mixed projection shape produced by an ordinary multi-Agent copy installation. Receipt, Manifest, and Sum publication uses a shared transaction lock and a recoverable journal whose deletion is the metadata commit point. The resulting targets use the ordinary copy-installation health, update, repair, and removal semantics rather than a separate adoption lifecycle.

An External Installation not represented by a selected supported lock is skipped. Invalid or unsupported-provider lock records and missing filesystem targets are also skipped independently rather than converted into Local Skills. One user confirmation authorizes only the explicitly selected scope subset of the preflight plan. Execution revalidates each planned candidate before its first authoritative write, skips a changed or missing candidate independently, and does not silently include candidates that appeared after preflight. The result reports taken-over and skipped counts without requiring per-Skill decisions.
