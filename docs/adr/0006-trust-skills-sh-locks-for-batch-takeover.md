# Trust skills.sh locks for batch takeover

SkillsGo offers one Library Batch Takeover action for currently discovered External Installations. The selected Library location is the authorization and scan boundary: All Skills selects User Scope and every accessible Added Project, Global selects only User Scope, and a Project route selects only that Workspace Scope. The App sends these scopes explicitly, and the CLI reads only their supported skills.sh locks. A lock supplies the candidate Repository identity, immutable version, and selected member; it does not by itself prove that the installed bytes are authentic. The CLI therefore resolves the immutable Repository Artifact, verifies the candidate member against those bytes, and rejects a mismatch before any authoritative write.

The Library runs this validation before confirmation as an independent CLI preflight and exposes the result to App callers as a Batch Takeover Plan. The plan returns one exact eligible count for All Skills, User Scope, and every accessible Added Project, plus an opaque ID bound to the validated candidate paths, normalized lock identities, immutable Repository coordinates, and filesystem-state digests. One eligible item means one physical Skill group within one declaration scope, so All Skills equals User Scope plus the per-Project counts and the confirmed count uses the same unit as the execution result. Preflight does not write `skillsgo.yaml`, `skillsgo-lock.yaml`, Scope Vendor, Repository Projections, or any Agent target; its only local output is the ephemeral authorization plan under the SkillsGo cache. Plans expire after 30 minutes, and cleanup retains at most 32 current plans including the new preflight, so repeated background validation cannot grow the cache without bound.

User data preservation and SkillsGo metadata integrity take precedence over making those installations uniform. Execution revalidates the planned path, installs the complete verified Repository through the ordinary Dependency, Lock, Scope Vendor, and Repository Projection transaction, and only after that transaction commits moves the superseded External directory to recoverable system trash. It never captures a private per-Skill artifact, creates a local-only version, writes an Installation Receipt, or overwrites the candidate in place. The resulting installation is indistinguishable from an ordinary Repository installation and uses the same inventory, verification, update, conflict, and selected-member removal semantics.

An External Installation not represented by a selected supported lock is skipped. Invalid or unsupported-provider lock records, unavailable immutable Repository Artifacts, content mismatches, and missing filesystem targets are also skipped independently rather than converted into Local Skills. One user confirmation authorizes only the explicitly selected scope subset of the preflight plan. Execution skips a changed or missing candidate independently and does not silently include candidates that appeared after preflight. The result reports taken-over and skipped counts without requiring per-Skill decisions.
