# Trust skills.sh locks for batch takeover

SkillsGo offers one Library Batch Takeover action for currently discovered External Installations. The CLI first reads the supported skills.sh lock in the current Workspace and the supported user lock. An installed Skill represented by either lock is trusted and taken over directly without a Hub or content match. Because these locks record one source entry per Skill but do not persist copy mode or general installation paths, the CLI scans known Agent Skill directories and associates every discovered physical copy or symlink with the trusted lock entry.

User data preservation and SkillsGo metadata integrity take precedence over making those installations uniform. Batch Takeover captures each stable physical directory, computes both the normalized Hub Content Digest and a complete filesystem-state digest covering modes and empty directories, verifies the resulting immutable Store artifact and metadata, and then reuses the normal copy-installation commit path without materializing content back over the existing target. A lock supplies trusted source identity but never proves a Hub artifact version, so these digests define a captured local baseline rather than impersonating an unverified Hub version. Baselines deduplicate only when both content and recoverable filesystem state are identical, distinct copied directories remain independent Installation Targets even when they came from one lock entry, and filesystem aliases of one symlink-resolved directory form one physical target group. Receipt, Manifest, and Sum publication uses a shared transaction lock and a recoverable journal whose deletion is the metadata commit point. The resulting targets use the ordinary copy-installation health, update, repair, and removal semantics rather than a separate adoption lifecycle.

An External Installation not represented by either lock is eligible only when its complete Content Digest identifies exactly one immutable Hub Skill artifact. That exact match supplies both the public Skill ID and immutable Hub version, allowing the Store artifact, metadata, declaration, and Installation Receipt to agree without claiming an unverified version. Zero matches and ambiguous matches are skipped. Invalid or unsupported-provider lock records and missing filesystem targets are also skipped independently rather than converted into Local Skills. One user confirmation authorizes only this batch, and the result reports taken-over and skipped counts without requiring per-Skill decisions.
