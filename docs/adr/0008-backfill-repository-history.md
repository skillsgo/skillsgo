# Backfill Repository history through Hub administration

SkillsGo keeps ordinary CLI installation demand-driven: `skillsgo add` resolves and publishes only the requested Repository version and does not expose a history-backfill option. Repository History Backfill is instead an explicit Hub administration operation exposed through `POST /api/v1/admin/repository-backfills`, with a corresponding authenticated status read under the same resource. One bounded request accepts a non-empty, duplicate-free list of Repository IDs, validates the complete list before creating work, and returns one independently meaningful result per Repository. Each accepted Repository owns its own Backfill Run; the request is only a batch transport and does not combine their lifecycle, success, or failure.

A Backfill Run is durable River work that discovers every canonical semantic-version Tag in one Source Repository and reuses the ordinary Repository Publisher for each historical Repository snapshot. It does not enumerate every commit or branch. Active runs are deduplicated per Repository, including across overlapping batch requests. A later request starts an incremental run that skips successful versions, retries failed versions, and includes newly observed Tags. Status reads support the same multi-Repository administrative workflow while preserving per-Repository `queued`, `running`, `complete`, or `complete_with_errors` status outside River. Versions commit independently: one missing, invalid, moved, or conflicting Tag does not prevent other versions from becoming available, no run may overwrite an existing immutable version, and handlers remain idempotent under River's at-least-once execution.

Historical Publication preserves exact download, Content Match, takeover, and restoration capabilities, but it does not make a Skill absent from the current Repository Publication reappear in public discovery, rankings, or current-version presentation. Published version metadata, Content Digests, and complete ZIPs are retained. Artifact lifecycle management, cold storage, ZIP eviction and restoration, security auditing, per-version Backfill status, cross-job batching, and source-host concurrency controls are outside this decision and require evidence before separate design.

Hub administration routes use Basic Auth without forcing a public Hub's product and distribution routes behind authentication. Existing `SKILLSGO_HUB_BASIC_AUTH_USER` and `SKILLSGO_HUB_BASIC_AUTH_PASS` continue to protect the whole Hub and therefore also protect administration routes. When global credentials are absent, `SKILLSGO_HUB_ADMIN_AUTH_USER` and `SKILLSGO_HUB_ADMIN_AUTH_PASS` protect only `/api/v1/admin/**`; when neither complete credential pair is configured, administration routes are not registered. A partial credential pair is a startup error. If both complete pairs are configured, global credentials take precedence, Hub starts normally, and it logs a warning that the Admin credentials are ignored.
